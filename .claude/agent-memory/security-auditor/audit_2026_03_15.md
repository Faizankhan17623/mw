## Key Security Patterns Found

### Critical
- backend/.env contains ALL production secrets (DB creds, Cloudinary, Razorpay, JWT key, email creds) - not in .gitignore but not currently tracked by git
- frontends/.env IS tracked by git (contains Razorpay test key ID)
- JWT secret is weak: "Faizan khan" (dictionary words)
- JWT token stored in localStorage (authSlice.js line 5, Auth.js line 268) - XSS-stealable
- OTP returned in API response (CreateOtp, line 164: `data:generate`)
- Password reset token returned in API response (Resetpassword.js line 42)
- User creation returns full user object including hashed password (Createuser.js line 105)
- No Helmet.js - zero HTTP security headers
- No express-mongo-sanitize - NoSQL injection possible on all endpoints
- No logout endpoint on backend - token never invalidated server-side
- Multiple unauthenticated routes that should require auth: FindUserNames, FindLoginEmail, FindNumber, FindCreationEmail, Theatre-info, Create-Theatre, Create-Orgainezer

### High
- FindNumber returns full user object including hashed passwords (User.js line 114: `data:user`)
- FindCreationEmail returns full user object (User.js line 147: `data:user`)
- CurrentLoginUser returns full user document without field filtering (Createuser.js line 489)
- CreateTheatrere returns full user creation data including hashed password (Theatrer.js line 80)
- Payment model uses String type for userid/theatreid/showid instead of ObjectId (BUG-5 confirmed)
- deleteComment has no ownership check - any Viewer can delete any comment
- User-supplied regex in SearchAll could cause ReDoS (Showlist.js line 133)
- ResetPassword doesn't clear token after use (only sets resetPasswordExpires to null, token field persists)
- CORS allows origin mismatch: index.js allows "mw-bay.vercel.app" but deployment is "mw-mocha.vercel.app"

### Medium
- No rate limiting on: password reset, payment, feedback, bug report, organizer login
- Cookie SameSite set to 'Lax' not 'Strict'
- Bug report file upload has no MIME type validation (BugReport.js)
- No CSRF protection
- Cloudinary uploads use resource_type: "auto" with no server-side file type validation in imageUploader.js

**How to apply:** Use these findings to prioritize remediation. Critical items should block deployment.
